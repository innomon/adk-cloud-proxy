# Authentication Token Implementation

This document describes the end-to-end implementation of authentication tokens within the ADK Cloud Proxy, including the "Circle of Trust" between the Router Proxy, OpenAI Proxy, and Connectors.

## 1. Token Format (NATS JWT)
The system uses [NATS NKey](https://github.com/nats-io/nkeys) (Ed25519) signed JWTs for all internal and client-to-proxy communication.

### Claims Structure
Tokens generated by `pkg/auth/auth.go` include the following claims:
- **iss (Issuer):** The public key of the entity that signed the token.
- **sub (Subject):** The public key of the entity identifying itself.
- **data.userid:** The identity of the end-user.
- **data.appid:** The target application ID.
- **data.sessionid:** (Optional) The specific ADK session ID.

## 2. Circle of Trust

### Component Roles
| Component | Role | Secret/Key Used | Source |
| :--- | :--- | :--- | :--- |
| **OpenAI Proxy** | Token Creator | `ISSUER_SEED` | Env Var |
| **Connector** | Token Creator | `NKEY_SEED` | Env Var |
| **Router Proxy** | Token Validator | `ISSUER_PUBLIC_KEY` | Env Var |

### Authentication Flows

#### A. Standard ADK Client Flow
1. Client provides a pre-signed NATS JWT in the `Authorization` header.
2. `router-proxy` validates the signature using `ISSUER_PUBLIC_KEY`.
3. `router-proxy` ensures the `appid` in the JWT matches the requested resource.

#### B. OpenAI Translation Flow
1. **External Authentication:** The OpenAI client must provide a Bearer token in the `Authorization` header matching the configured `openai.api_key`.
2. **Identity Synthesis:** If authenticated, the OpenAI Proxy uses its `ISSUER_SEED` to generate a "trusted" NATS JWT for the `openai-user`.
3. **Internal Auth:** The Proxy calls the local ADK routes using this generated token.
4. **Validation:** The `router-proxy` accepts the request because the token is signed by the trusted `ISSUER_SEED`.

## 3. Configuration

### Environment Variables
- `ISSUER_PUBLIC_KEY`: Public key for NATS JWT validation.
- `ISSUER_SEED`: Seed used by OpenAI Proxy to sign internal tokens.
- `OPENAI_API_KEY`: (Optional) Static API key required for external OpenAI clients.

### config.yaml
```yaml
openai:
  api_key: "sk-..."
  default_app_id: "ollama"
  default_user_id: "openai-user"
```


#### C. Connector Stream Flow
1. The Connector (behind firewall) generates a JWT using its `NKEY_SEED`.
2. It sends this token in the gRPC metadata during the `Connect` call.
3. `router-proxy` validates the token to register the Connector for a specific `appID`.

## 3. Implementation Details

### Generation
`pkg/auth/auth.go`:
```go
func GenerateToken(seed []byte, userID, appID, sessionID string, ttl time.Duration) (string, error)
```
Uses `nkeys.FromSeed(seed)` to sign a `jwt.GenericClaims` object.

### Validation
`pkg/auth/auth.go`:
```go
func (v *Validator) Validate(tokenStr string) (*Claims, error)
```
Uses `jwt.DecodeGeneric(tokenStr)` to verify the Ed25519 signature and check the `iss` claim against the configured `ISSUER_PUBLIC_KEY`.

### Multi-Auth Support
`pkg/auth/dual_validator.go` allows the system to support both:
1. **NATS JWTs:** (Hand-crafted, used by Connectors and OpenAI Proxy).
2. **OAuth JWTs:** (EdDSA signed, used by external SPA/WhatsApp clients).

## 4. Security Considerations
- **ISSUER_SEED Protection:** The `ISSUER_SEED` allows full impersonation of any user for any app. It must be stored securely (e.g., Secret Manager) and never logged.
- **Token Expiry:** Tokens are typically generated with a 1-hour TTL.
- **OpenAI Endpoint:** The `/v1` endpoint is currently open (unauthenticated) but generates authenticated internal traffic. Access control should be handled at the Cloud Run/Ingress level if needed.
